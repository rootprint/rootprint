```mdx
## Start with the built-in OpenTelemetry index

If you deploy Logwiz with the default project setup, start with `otel-logs-v0_9`. It is already wired into the bundled Quickwit setup, so you can confirm the full ingest path before you introduce custom indexes.

## How Logwiz reads OpenTelemetry fields

When Logwiz syncs an index whose ID starts with `otel-logs-`, it applies these defaults on first sync:

| Logwiz field    | Default OTel path                 |
| --------------- | --------------------------------- |
| Level field     | `severity_text`                   |
| Message field   | `body`                            |
| Traceback field | `attributes.exception.stacktrace` |

All field settings accept dot notation. If your payload keeps the human-readable message in a nested key such as `body.message`, update the message field in [Per Index configuration](/configuration/index-config).
```

- [ ] **Step 2: Write the HTTP ingestion workflow and example**

````mdx
## Send logs over HTTP

<Steps>
  <Step title="Create an ingest token">
    Open **Administration -> Send Logs**, create a token, and copy it before closing the dialog. The token value is shown only once.
  </Step>

<Step title="Choose the index you want to write to">
	Use `otel-logs-v0_9` for the default OpenTelemetry flow, or replace it with another Quickwit index
	ID if you already manage your own indexes.
</Step>

  <Step title="Post logs to the Logwiz gateway">
    Send requests to `/api/ingest/{indexId}` on your Logwiz instance.

    ```bash
    curl -X POST 'https://logs.example.com/api/ingest/otel-logs-v0_9?commit=wait_for' \
      -H 'Authorization: Bearer <your-ingest-token>' \
      -H 'Content-Type: application/x-ndjson' \
      --data-binary @- <<'EOF'
    {"timestamp_nanos":1710000000000000000,"severity_text":"INFO","body":"User logged in","service":"frontend"}
    {"timestamp_nanos":1710000060000000000,"severity_text":"ERROR","body":"Checkout failed","attributes":{"exception":{"stacktrace":"Traceback (most recent call last): example"}}}
    EOF
    ```

  </Step>
</Steps>

Logwiz authenticates the request, checks the token's index allowlist, and forwards the request to Quickwit. Logwiz does not rewrite the request body, so send the format your Quickwit ingest endpoint expects. In the default setup, that means NDJSON with `Content-Type: application/x-ndjson`.
````

Ingest endpoints require a bearer token in the `Authorization` header.

```http
Authorization: Bearer <your-ingest-token>
```

Create and revoke ingest tokens in the Logwiz admin panel at **Administration -> Send Logs**. Each token can optionally be scoped to one or more index IDs.
